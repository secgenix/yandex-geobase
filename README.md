## Техническое задание

### Наименование проекта

Сервис отображения геоданных на базе Яндекс.Карт.

### Цель проекта

Разработать веб-приложение, отображающее интерактивную карту с маркерами, координаты и атрибуты которых загружаются из базы данных, с возможностью поиска, фильтрации и просмотра подробной информации по объектам. FastAPI подходит для построения backend API, а JavaScript API Яндекс.Карт v3 позволяет создавать карту, добавлять слои, источники данных и маркеры в браузере.[^2][^1]

## Назначение системы

Система предназначена для:

- отображения объектов на карте по данным из БД;
- поиска объектов по текстовым полям;
- фильтрации объектов по атрибутам;
- просмотра карточки объекта;
- масштабирования, перемещения карты и работы с кластерами при большом количестве маркеров. API Яндекс.Карт v3 поддерживает карту как контейнер `YMap`, стандартный слой схемы `YMapDefaultSchemeLayer`, пользовательские источники геоданных `YMapFeatureDataSource` и маркеры `YMapMarker`, что соответствует задачам данного проекта.[^1]


## Технологический стек

| Компонент | Технология | Назначение |
| :-- | :-- | :-- |
| Backend | FastAPI | REST API для выдачи геоданных и фильтров [^2] |
| Работа с БД | asyncpg | Асинхронное взаимодействие с PostgreSQL [^2] |
| База данных | PostgreSQL | Хранение объектов и координат |
| Frontend | HTML/CSS/JavaScript | Пользовательский веб-интерфейс |
| Карта | Yandex Maps JS API v3 (`ymaps3`) | Отображение карты, маркеров, слоев [^3][^1] |
| Контейнеризация | Docker, Docker Compose | Развертывание приложения |
| Web-сервер | Nginx, опционально | Раздача статики и проксирование API |

## Общая архитектура

Система должна состоять из следующих компонентов:

1. Frontend-клиент, отображающий карту и элементы управления.
2. Backend-сервис на FastAPI, предоставляющий REST API.
3. PostgreSQL для хранения объектов.
4. Docker-окружение для локального и серверного запуска.

Взаимодействие предполагается по схеме: браузер запрашивает список объектов и фильтров у FastAPI, backend получает данные из PostgreSQL через asyncpg и возвращает JSON, после чего frontend создает или обновляет маркеры на карте. Такой подход соответствует модели работы `YMap` и динамическому добавлению дочерних сущностей через `addChild()` в JS API v3.[^2][^1]

## Функциональные требования

### Карта

В интерфейсе должна отображаться интерактивная карта Яндекс.Карт:

- начальный центр и масштаб задаются конфигурацией;
- поддерживаются zoom, pan;
- должна использоваться схема карты через `YMapDefaultSchemeLayer`;
- контейнер карты должен занимать основную область экрана. Базовый сценарий инициализации карты в API v3 строится на ожидании `ymaps3.ready`, создании `new YMap(...)` и добавлении слоев через `addChild()`.[^3][^1]


### Отображение объектов

На карту должны наноситься маркеры объектов из базы данных. Для каждого объекта должны поддерживаться:

- идентификатор;
- наименование;
- координаты;
- категория;
- статус;
- описание;
- дополнительные атрибуты по бизнес-логике.

Для визуализации допускаются два режима:

- отдельные DOM-маркеры через `YMapMarker`;
- работа через источник `YMapFeatureDataSource` и слой `YMapLayer` для feature/marker-слоя. API v3 явно поддерживает и `YMapMarker`, и `YMapFeatureDataSource` с `YMapLayer`, что удобно для разной плотности точек и дальнейшего расширения.[^1]


### Кластеризация

При большом количестве объектов должна использоваться кластеризация маркеров. Для этого необходимо предусмотреть подключение пакета `@yandex/ymaps3-clusterer` и алгоритма `clusterByGrid`, как это показано в примерах использования API v3.[^1]

### Поиск

Должен быть реализован текстовый поиск по объектам:

- по названию;
- по адресу, если поле предусмотрено;
- по внешнему идентификатору, если требуется.

Поиск должен работать без полной перезагрузки страницы, с повторным запросом данных через API.

### Фильтрация

Необходимо реализовать фильтрацию по одному или нескольким параметрам:

- категория;
- статус;
- регион/город;
- пользовательские признаки.

Фильтры должны комбинироваться между собой. После изменения фильтра список объектов на карте должен обновляться асинхронно.

### Карточка объекта

При клике на маркер пользователь должен видеть карточку объекта с подробной информацией:

- название;
- координаты;
- адрес;
- категория;
- статус;
- описание;
- дополнительные поля.

Карточка может отображаться в popup, боковой панели или отдельном блоке справа от карты.

### Список объектов

Дополнительно необходимо предусмотреть список найденных объектов рядом с картой или под картой:

- синхронизация списка с фильтрами;
- выделение объекта в списке при клике на маркер;
- центрирование карты на объекте при выборе в списке.


## Нефункциональные требования

### Производительность

- Время ответа API на выборку объектов: не более 1 секунды при типовом объеме данных.
- Отображение карты и первичная загрузка объектов: не более 3 секунд при нормальном сетевом соединении.
- При большом количестве точек должна применяться кластеризация и/или выдача объектов по bounds карты.

Практики асинхронной работы FastAPI с PostgreSQL предполагают использование connection pool и аккуратное управление жизненным циклом подключений, что важно для стабильной производительности.[^2]

### Масштабируемость

Система должна допускать:

- увеличение количества объектов;
- добавление новых фильтров;
- расширение API;
- горизонтальное масштабирование контейнеров backend.


### Надежность

- Обработка ошибок API с понятными сообщениями.
- Логирование ошибок backend.
- Валидация входных параметров.
- Корректная работа при отсутствии данных.


### Безопасность

- Валидация query-параметров API.
- Параметризованные SQL-запросы.
- Хранение секретов и DSN в переменных окружения.
- Ограничение CORS по списку разрешенных доменов.
- API-ключ Яндекс.Карт не должен быть захардкожен в исходниках; в примерах Яндекса ключ передается в URL подключения API как `apikey=<YOUR_APIKEY>`.[^3]


## Требования к backend

### Стек backend

Backend должен быть реализован на Python с использованием FastAPI и asyncpg. Для FastAPI с PostgreSQL рекомендуется асинхронная работа с пулом соединений и явное управление транзакциями/жизненным циклом соединений.[^2]

### Основные задачи backend

Backend должен:

- предоставлять REST API для карты;
- возвращать список объектов по фильтрам;
- возвращать один объект по ID;
- возвращать справочники для фильтров;
- поддерживать пагинацию или ограничение выборки при необходимости;
- поддерживать фильтрацию по bounds карты как опциональную оптимизацию.


### Предлагаемые API-методы

#### `GET /api/v1/health`

Проверка работоспособности сервиса.

Ответ:

```json
{
  "status": "ok"
}
```


#### `GET /api/v1/objects`

Получение списка объектов.

Параметры:

- `search`: строка поиска
- `category`: категория
- `status`: статус
- `city`: город/регион
- `bbox`: границы карты в формате `minLon,minLat,maxLon,maxLat`
- `limit`: ограничение количества записей
- `offset`: смещение

Пример ответа:

```json
{
  "items": [
    {
      "id": 1,
      "name": "Объект 1",
      "latitude": 55.75,
      "longitude": 37.61,
      "category": "office",
      "status": "active",
      "address": "Москва, ...",
      "description": "Описание"
    }
  ],
  "total": 1
}
```


#### `GET /api/v1/objects/{id}`

Получение детальной информации по объекту.

#### `GET /api/v1/filters`

Получение справочников для фильтров:

```json
{
  "categories": ["office", "store"],
  "statuses": ["active", "inactive"],
  "cities": ["Москва", "Новосибирск"]
}
```


## Требования к БД

### СУБД

PostgreSQL.

### Структура основной таблицы

Рекомендуемая таблица `geo_objects`:

```sql
CREATE TABLE geo_objects (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    address TEXT,
    description TEXT,
    category VARCHAR(100),
    status VARCHAR(50),
    city VARCHAR(100),
    latitude DOUBLE PRECISION NOT NULL,
    longitude DOUBLE PRECISION NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```


### Индексация

Необходимо создать индексы:

- по `category`;
- по `status`;
- по `city`;
- по `name` для поиска;
- составные индексы по наиболее частым фильтрам.

Если планируется геопоиск и выборка по области карты на больших объемах данных, желательно отдельно рассмотреть расширение PostGIS, хотя для базовой версии проекта это не является обязательным требованием.

## Требования к frontend

### Общие требования

Frontend должен быть реализован как SPA-подобный интерфейс без полной перезагрузки страницы. Карта должна инициализироваться через Yandex Maps JS API v3, который подключается отдельным `script` с указанием `apikey`, после чего логика строится через `await ymaps3.ready`.[^3]

### Элементы интерфейса

На странице должны быть:

- область карты;
- строка поиска;
- панель фильтров;
- список объектов;
- зона отображения карточки выбранного объекта;
- индикатор загрузки;
- сообщение “ничего не найдено”.


### Логика работы карты

Frontend должен:

1. Инициализировать карту.
2. Запросить объекты через API.
3. Отрисовать маркеры.
4. Обновлять маркеры при изменении фильтров.
5. Обрабатывать клик по маркеру.
6. При необходимости выполнять кластеризацию.

API v3 поддерживает добавление дочерних сущностей на карту через `addChild`, а для пользовательских данных — `YMapFeatureDataSource` и `YMapLayer`; это нужно заложить в архитектуру фронтенда с переиспользованием слоя данных, а не полной пересборкой карты.[^1]

## Требования к логике backend

### Подключение к БД

При старте приложения должен создаваться пул соединений asyncpg, а при завершении — корректно закрываться. Практики работы FastAPI с async PostgreSQL рекомендуют pooling и управление жизненным циклом подключения через зависимости или startup/shutdown hooks.[^2]

### Формирование SQL-запросов

SQL-запросы должны:

- собираться динамически на основании фильтров;
- использовать параметризацию;
- поддерживать limit/offset;
- не допускать SQL injection.


### Формат выдачи координат

В API координаты должны возвращаться явно как longitude и latitude. На frontend необходимо соблюдать формат координат, ожидаемый Яндекс.Картами; в примерах API v3 используется массив координат в формате `[longitude, latitude]`.[^1]

## Требования к логике frontend

### Инициализация карты

Примерный сценарий:

- дождаться загрузки `ymaps3.ready`;
- создать `YMap`;
- добавить `YMapDefaultSchemeLayer`;
- добавить слой для маркеров или features;
- загрузить точки из backend.

Именно такой базовый паттерн показан в документации и справочнике API v3.[^3][^1]

### Работа с маркерами

Каждый маркер должен:

- иметь координаты;
- отображать визуальный элемент;
- содержать ID объекта;
- по клику инициировать открытие карточки.

`YMapMarker` предназначен для размещения DOM-элемента на карте, а для групповой работы с данными можно использовать `YMapCollection` или кластеризатор.[^1]

### Работа с кластеризацией

При количестве точек выше порогового значения, например 100+, должна автоматически включаться кластеризация. Для этого требуется использовать пакет `@yandex/ymaps3-clusterer` и стратегию `clusterByGrid`, которую Яндекс показывает в рабочих примерах.[^1]

## Пользовательские сценарии

### Сценарий 1

Пользователь открывает страницу, видит карту и все доступные объекты.

### Сценарий 2

Пользователь вводит текст в поиск, список и маркеры обновляются.

### Сценарий 3

Пользователь выбирает фильтр “категория = office”, на карте остаются только подходящие объекты.

### Сценарий 4

Пользователь кликает по маркеру, справа открывается карточка объекта.

### Сценарий 5

Пользователь отдаляет карту, объекты объединяются в кластеры.

## Критерии приемки

Проект считается реализованным, если:

- карта отображается в браузере;
- объекты загружаются из PostgreSQL;
- маркеры корректно отображаются на карте;
- поиск работает;
- фильтры работают;
- карточка объекта открывается;
- backend запускается в Docker;
- все сервисы поднимаются через Docker Compose;
- конфигурация через переменные окружения работает;
- API возвращает корректный JSON;
- при большом количестве объектов работает кластеризация. Возможность такой реализации технически поддерживается API Яндекс.Карт v3 через сущности карты, маркеры, data source и кластерный пакет.[^1]


## Ограничения и допущения

- В рамках базовой версии не реализуется авторизация, если она не требуется отдельно.
- В рамках базовой версии редактирование объектов через UI не входит в объем работ.
- API Яндекс.Карт требует действующий API-ключ для отображения карты.[^3]
- Для production-режима необходимо отдельно определить домен, SSL, reverse proxy и стратегию хранения секретов.
